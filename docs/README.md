# Documentation Index

Use this directory by role instead of reading every document end to end.

## If you are buying parts

- Start with `approved_sku_sheet.md` for the exact parts to purchase.
- Read `buying_guide.md` for the short buying path and prototype-versus-production notes.
- Use `validated_bom_v3.md` if you need the engineering rationale behind the current baseline.
- Read `parts_audit_best_possible.md` only when evaluating higher-cost production upgrades.

## If you are building the robot

- Start with `quick_start.md` for the shortest setup path.
- Use `wiring_guide.md` while assembling power, GNSS, relays, and Pixhawk I/O.
- Keep `troubleshooting.md` open during first power-on, bench checks, and field debugging.
- Read `failure_modes.md` before field trials to understand known hazards and fallback behavior.

## If you are operating or maintaining the robot

- Use `maintenance.md` for service intervals and inspection routines.
- Use `troubleshooting.md` for run-time failures, RTK issues, and paint-system faults.
- Read `failure_modes.md` for safety-critical behavior and recovery expectations.

## If you are deploying the software platform

- Start with `deployment.md` for Railway production deployment.

## Historical and analysis documents

- `bom.md` is retained as a historical BOM archive and should not be used as the final buying source.
- `business-plan.md`, `research_report.md`, `competitive_benchmarks.md`, and `overengineering_analysis.md` are supporting analysis documents, not build checklists.
